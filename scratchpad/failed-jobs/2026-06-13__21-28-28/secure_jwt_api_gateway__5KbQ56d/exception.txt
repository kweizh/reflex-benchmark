# Secure JWT API Gateway with Reflex `api_transformer`

## Background
Reflex serves its UI through a FastAPI backend. By passing a custom FastAPI application to the `api_transformer` parameter of `rx.App`, you can expose additional REST endpoints on the same backend port. In this task, you will mount a custom JWT authentication API into a Reflex app and wire a Reflex page that exercises that API from a State event handler. Sensitive material (the issued token) must never leak into the frontend bundle, so it must live in a backend-only state var.

## Requirements
- Build a single Reflex project that simultaneously serves:
  1. A custom FastAPI router mounted via `api_transformer` exposing the routes documented in *Acceptance Criteria*.
  2. A Reflex page at the root path (`/`) that demonstrates an end-to-end login flow against those routes.
- The authentication API must issue and validate JWTs using algorithm **HS256** with the secret value given in *Acceptance Criteria*. Use the `PyJWT` package (`import jwt`).
- The page must contain a button labelled `Login`. Clicking it must:
  1. Call the backend `/auth/login` endpoint from a Reflex event handler using `httpx` or `requests` (server-side HTTP call, not browser fetch).
  2. Store the returned JWT in a **backend-only** state var (the field name must start with an underscore so it is never serialized to the client).
  3. Use that stored JWT to call `/auth/me` (again from an event handler) and place the returned `username` into a normal (client-visible) state var.
- After a successful login, the page must render the authenticated username somewhere in the visible DOM.
- Manage the Python environment with `uv`. Initialize the Reflex project non-interactively with `uv run reflex init --template blank`.

## Implementation Hints
- The `api_transformer` parameter of `rx.App` accepts a `FastAPI` instance directly; Reflex will mount its own internal routes alongside yours on port 8000.
- Backend-only state vars are conventional Python attributes on an `rx.State` subclass whose name begins with `_`. They are never sent to the client.
- The Reflex backend itself listens on `http://localhost:8000`, so server-side HTTP calls from event handlers can target that base URL.
- PyJWT's HS256 flow is essentially `jwt.encode(payload, secret, algorithm="HS256")` / `jwt.decode(token, secret, algorithms=["HS256"])`.
- The JWT secret, username, and password are fixed values supplied in *Acceptance Criteria* — hard-code them into the project. Do **not** read them from environment variables.

## Acceptance Criteria
- Project path: `/home/user/secure_jwt_api_gateway`
- Python environment is managed by `uv` (i.e. `pyproject.toml` exists and `uv run` works at the project root). `reflex` and `PyJWT` must be installable dependencies of the project.
- Start command: `cd /home/user/secure_jwt_api_gateway && uv run reflex run --loglevel info` (the executor must ensure this command brings up both the frontend on port 3000 and the backend — including the mounted custom routes — on port 8000).
- Frontend port: `3000`
- Backend port: `8000`
- **Fixed credentials and secret** (use these exact values; they are randomly generated per task instance):
  - Username: `alice_h4k9m2`
  - Password: `P@ssw0rd_X9zL2qN8`
  - JWT signing secret (HS256): `9f3e8a1c4b7d2e5f8a0b3c6d9e2f1a4b7c0d3e6f9a2b5c8d1e4f7a0b3c6d9e2f`
- HTTP API (served on port 8000, mounted via `api_transformer`):
  - `POST /auth/login`
    - Request JSON body: `{"username": string, "password": string}`
    - On valid credentials: returns HTTP 200 with JSON body `{"access_token": string, "token_type": "bearer"}`. The `access_token` MUST be a JWT signed with HS256 using the secret above and MUST contain a `sub` claim whose value equals the supplied username.
    - On invalid credentials: returns HTTP 401.
  - `GET /auth/me`
    - Requires header `Authorization: Bearer <token>`.
    - On a valid token signed with the secret above: returns HTTP 200 with JSON body `{"username": string}` where `username` matches the `sub` claim of the token.
    - On missing or invalid token: returns HTTP 401.

  - The page contains a clickable element whose visible text is exactly `Login`.
  - Clicking `Login` triggers a state event handler that performs the login + identity round-trip described in *Requirements*.
  - After a successful login, the page renders the authenticated username (`alice_h4k9m2`) somewhere in the DOM. Before clicking Login, that username string must NOT appear in the initial server-rendered HTML of the root page.

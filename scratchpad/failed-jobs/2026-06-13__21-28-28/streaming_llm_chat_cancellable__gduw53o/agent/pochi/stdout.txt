# Cancellable Streaming LLM Chat Panel (Reflex) ## Background You are building a small Reflex application that simulates an LLM chat experience with **mid-response cancellation**. The UI is a chat panel; submitting a prompt streams a simulated assistant reply token-by-token, while a **Stop** control can interrupt the stream between any two tokens. A loading spinner is bound to a backend-only flag while the stream is running. The interesting part is on the server: the streaming work runs in a Reflex background event, holds the state lock only when mutating shared state, and cooperatively observes a cancellation flag. ## Requirements * Reflex application at `/home/user/streaming_chat` initialized with the `blank` template and managed by `uv`. * A `StreamingChatState` class (a subclass of `rx.State`) that exposes: * The currently rendered assistant response, * A loading indicator surfaced through a Reflex computed/base var, * Backend-only flags (names starting with `_`) for the in-flight streaming status and the cooperative cancel signal. * An async event handler decorated with `@rx.event(background=True)` that simulates a streaming LLM reply token-by-token. The handler must: * Acquire the state lock with `async with self` **per chunk** to publish progress safely, * Sleep a small bounded amount between chunks (between 50 ms and 250 ms), * Check the backend-only cancel flag between yields and stop cleanly when it is set, * Emit at least 12 chunks for a normal (uninterrupted) reply, * Reset the streaming flag to `False` deterministically on both normal completion and cancellation, * Never mutate shared state outside an `async with self` block (must not raise `ImmutableStateError`). * A regular (non-background) event handler that flips the cancel flag when the **Stop** button is pressed. * A chat UI page on `/` containing at minimum: a prompt input, a **Send** button, a **Stop** button, the assistant response, and a loading indicator whose visibility is driven by the backend-only streaming flag. * A FastAPI router mounted via `rx.App(api_transformer=...)` that exposes a small, stable HTTP contract for external observation and control of the streaming task (see *Acceptance Criteria*). The Reflex background event and the FastAPI endpoints **must share the same underlying streaming controller** so that triggering the stream from the API exercises the same cancellation/lock semantics that the UI does. * The server is started by the verifier in **backend-only** mode on port 8000. * After the test runs, **all background servers must be terminated** (the verifier starts its own server; leftover servers will cause port conflicts). ## Implementation Hints * The friction you should design around is `ImmutableStateError`: a background task cannot write `self.<var>` outside `async with self`. Plan your loop so the per-chunk update is the only thing inside the lock. * Backend-only vars (names starting with `_`) are *not* serialized to the client; if you want to drive the spinner from `_is_streaming`, surface it via a regular base var or a computed var. * Because Reflex State is per-session and the verifier reaches the app via HTTP (not Socket.IO), introduce a module-level controller object that holds the live streaming status and is mutated by the background event. Both the FastAPI endpoints and the Reflex background event delegate to that controller, while the rx.State still owns the per-chunk `async with self` updates and the backend-only flags. * Use `uv` to add dependencies (`uv add reflex 'fastapi'`); the verifier runs the app with `uv run reflex run --backend-only`. * Keep server logs in a file so an `ImmutableStateError` (if any) becomes observable to the verifier. ## Acceptance Criteria * Project path: `/home/user/streaming_chat` * Start command: `cd /home/user/streaming_chat && nohup uv run reflex run --backend-only --backend-port 8000 --loglevel debug > /home/user/streaming_chat/server.log 2>&1 &` * Port: `8000` * Server log: `/home/user/streaming_chat/server.log` (must exist after the server has been started) * Source code requirements (verified by inspecting the project source tree under `/home/user/streaming_chat`): * At least one Python file contains the literal substring `@rx.event(background=True)`. * At least one Python file contains the literal substring `async with self`. * At least one Python file declares a backend-only streaming flag whose identifier matches `_is_streaming`. * At least one Python file declares a backend-only cancel flag whose identifier matches `_should_cancel`. * HTTP API (mounted via `api_transformer`), all responses JSON with HTTP 200 unless noted: * `POST /api/chat/send` * Request body: ```json {"prompt": string} ``` * Response: ```json {"accepted": true} ``` * Side effect: starts (or restarts) the cancellable streaming background task. If a previous stream was running, it must be cancelled and superseded. * `POST /api/chat/cancel` * Request body: empty or `{}`. * Response: ```json {"cancelled": true} ``` * Side effect: sets the cooperative cancel flag observed by the background task between yields. * `GET /api/chat/status` * Response shape: ```json { "is_streaming": boolean, "current_response": string, "chunks_streamed": number, "was_cancelled": boolean, "completed": boolean } ``` * Field semantics: * `is_streaming` is `true` while the background task is producing chunks and `false` otherwise. * `current_response` is the assistant text accumulated so far. * `chunks_streamed` is the count of chunks published so far for the current run. * `was_cancelled` is `true` iff the most recent run was stopped via the cancel flag before its natural end. * `completed` is `true` iff the most recent run reached its natural end (no cancellation) and is no longer streaming. * Behavioral requirements (verifier will exercise these via HTTP): * **Uninterrupted run:** after `POST /api/chat/send`, within 15 s `GET /api/chat/status` must report `completed=true`, `was_cancelled=false`, `is_streaming=false`, and `chunks_streamed >= 12`. * **Cancelled run:** if `POST /api/chat/cancel` is issued mid-stream, within 1 s `GET /api/chat/status` must report `is_streaming=false` and `was_cancelled=true`, and `chunks_streamed` must not increase by more than 2 after the cancel call. * **Loading transition:** during an uninterrupted run, the verifier observes `is_streaming=true` at least once and `is_streaming=false` after completion (deterministic True→False transition). * Server log requirements: * `/home/user/streaming_chat/server.log` must NOT contain the literal substring `ImmutableStateError` after either run. * Cleanup requirements: * The agent must terminate any background Reflex servers (for example with `pkill -f 'reflex run'`) before declaring the task complete.

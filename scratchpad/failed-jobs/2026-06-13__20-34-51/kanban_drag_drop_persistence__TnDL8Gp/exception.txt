# Kanban Board with Backend Persistence and Move API

## Background
Reflex serves a single ASGI app that combines a Next.js-compiled frontend with a FastAPI backend. By passing a custom `FastAPI` instance to `rx.App(api_transformer=...)`, you can mount additional REST endpoints onto the same backend port that drives Reflex events. Reflex's database layer wraps SQLModel/SQLAlchemy through `rx.Model`; transactional work runs inside `rx.session()`.

In this task you will build a three-column Kanban board (TODO / DOING / DONE) whose card order is fully persisted in SQLite. A custom FastAPI route exposes the move operation as a server-side API so that any client (the Reflex UI, curl, or the verifier) can deterministically reorder cards. Card positions must remain 0-dense per column after every move.

## Requirements
- Build a single Reflex project that serves both:
  1. A Reflex page at the root path (`/`) that renders three columns (`TODO`, `DOING`, `DONE`) using `rx.foreach`, with cards inside each column displayed in ascending `position` order.
  2. A custom FastAPI app mounted via `rx.App(api_transformer=...)` that exposes `POST /api/cards/move`.
- Persist cards in SQLite using `rx.Model` with the schema described in *Acceptance Criteria*.
- Seed the database on app startup (if empty) with the exact set of cards described in *Acceptance Criteria*.
- The move endpoint must run as a single DB transaction and normalise positions so that, after every successful move, each of the three columns contains a 0-dense sequence of positions (`0, 1, 2, ...`).
- The Reflex page must read the current cards from the same database and render them grouped by column, ordered by `position`. Use `rx.foreach` to iterate over both columns and cards.
- Manage the Python environment with `uv` and initialise the Reflex project non-interactively (see plan.md).

## Implementation Hints
- `rx.App` accepts an `api_transformer` parameter. Passing a `FastAPI()` instance mounts your routes alongside Reflex's reserved `/ping` and `/_event` routes on port 8000.
- `rx.Model` is a thin wrapper around `SQLModel`. Use `rx.session()` (sync) inside the FastAPI route handler for the move transaction.
- After moving a card into a target column, you typically (a) remove it from its old column, (b) insert it at the target index, then (c) renumber positions inside *both* the source and destination columns so they are contiguous starting at 0.
- To make the page reflect the DB state on reload, load cards from the DB in an `on_load` event handler (or in a state initializer) and store them in a state var. `rx.foreach` over a `list[Card]` works because `rx.Model` instances are serializable.
- Reflex's database migration commands are `uv run reflex db init` / `makemigrations` / `migrate`. Running them once after defining the `Card` model is enough.
- Use `rx.foreach` (not Python `for`) when rendering state-bound lists; nested `rx.foreach` is allowed.

  - `id` (integer primary key, auto-generated)
  - `title` (text, non-null)
  - `column` (text, non-null; one of `TODO`, `DOING`, `DONE`)
  - `position` (integer, non-null; 0-based index inside its column)

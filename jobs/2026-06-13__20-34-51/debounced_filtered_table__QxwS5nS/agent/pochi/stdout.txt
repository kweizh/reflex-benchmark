# Debounced Multi-Filter Data Table (Reflex) ## Background Build a Reflex web application that renders a searchable, filterable, sortable data table backed by a SQLite database. The table lists a catalog of `Product` rows that the application must seed deterministically at startup. The text-search filter must be debounced on the client (so rapid typing produces a single query), and the recomputation of filter results must run as an async background event using SQLModel `where`/`order_by` clauses against `rx.asession`. Use Astral `uv` to manage the Python environment and to drive every Reflex CLI command, per the deployment plan for this benchmark. ## Requirements * Initialize a Reflex project named `filtered_table` at `/home/user/filtered_table` using the blank template. * Define a SQLModel-backed table `Product` (subclass of `rx.Model`) with at minimum these columns: * `id` (auto-increment primary key) * `name: str` * `category: str` * `sku: str` * `price: float` * `in_stock: bool` * Seed the database with EXACTLY 240 rows the first time the app starts, following the deterministic algorithm in the "Seed Algorithm" sub-section below. Subsequent starts must not duplicate rows. * Build a single page mounted at `/` containing the following reactive filter controls bound to State vars: * Text search input wrapped in `rx.debounce_input(...)` with `debounce_timeout=300` (milliseconds). * A category dropdown / select offering the literal value `All` plus each of the six category names. * Two numeric inputs for `min_price` and `max_price`. * A checkbox labeled "In stock only" bound to a boolean state var. * A sort-column selector and a sort-direction (asc/desc) selector. * The State must expose at least the following synchronized base vars: `result_count: int`, `filtered: list[Product]`, and the individual filter/sort vars listed above. * Recomputation of `filtered` and `result_count` MUST happen inside an `@rx.event(background=True)` async handler that opens an `rx.asession()` and constructs a SQLModel `select` with `.where(...)` clauses and an `.order_by(...)` clause. All mutations to state must happen inside an `async with self:` block. The background event must be triggered whenever any filter or sort var changes. * The results table must be rendered with `rx.foreach` iterating over `State.filtered`, displaying at least `id`, `name`, `category`, `price`, and `in_stock` per row. Render `result_count` somewhere visible above or below the table. * Additionally expose a plain JSON HTTP endpoint `GET /api/filter` mounted via the Reflex `api_transformer` (a FastAPI app). This endpoint applies the exact same filter+sort semantics against the same SQLite database and returns a JSON object. The endpoint exists so that the filter logic can be exercised programmatically. See the "API Contract" sub-section. ### Seed Algorithm (must be implemented exactly) Use this list, in this order, as the only category values: ``` CATEGORIES = ["Electronics", "Books", "Clothing", "Home", "Toys", "Sports"] ``` For each category index `c` in `0..5` (inclusive), and for each item index `i` in `0..39` (inclusive), insert a row with: * `name = f"{CATEGORIES[c]} #{i+1:02d}"` (zero-padded width 2; e.g. `Books #07`, `Sports #40`) * `category = CATEGORIES[c]` * `sku = f"{CATEGORIES[c][:3].upper()}-{i+1:03d}"` (e.g. `BOO-007`, `SPO-040`) * `price = round(5.0 + (c * 5) + (i * 1.0), 2)` (a float; e.g. Electronics #01 = 5.0, Sports #40 = 69.0) * `in_stock = (i % 4) != 3` (true for 30 of every 40 rows in each category) The insertion order MUST be category outer loop, item inner loop, so primary keys are assigned 1..240 with `Electronics #01` = id 1 and `Sports #40` = id 240. Total rows after seeding: 240. Idempotency: if the `product` table already contains rows when the application starts, do NOT re-seed. ### Filter & Sort Semantics (used by both the State background event and `/api/filter`) * `search` (string, default ""): case-insensitive substring match against `Product.name`. Empty string means no constraint. * `category` (string, default "All"): when value equals `All` or empty string, no constraint; otherwise equality match against `Product.category`. * `min_price` (float or null, default null): when provided, `Product.price >= min_price`. * `max_price` (float or null, default null): when provided, `Product.price <= max_price`. * `in_stock_only` (bool, default false): when true, restrict to rows with `in_stock == true`. * `sort_by` (string, default `id`): one of `id`, `name`, `price`, `category`. * `sort_dir` (string, default `asc`): one of `asc`, `desc`. ### API Contract for `GET /api/filter` Query parameters (all optional): * `search`: string * `category`: string * `min_price`: float * `max_price`: float * `in_stock_only`: `true` or `false` (case-insensitive) * `sort_by`: `id` | `name` | `price` | `category` * `sort_dir`: `asc` | `desc` Response: `200 OK` with JSON body shaped exactly like: ``` { "result_count": , "filtered": [ { "id": <int>, "name": <str>, "category": <str>, "sku": <str>, "price": <float>, "in_stock": <bool> }, ... ] } ``` The length of `filtered` MUST equal `result_count`. Sort order in the JSON list MUST match the requested sort. ## Implementation Hints * Per the plan, manage the Python environment with `uv`. Initialize via `uv init` and `uv add reflex`, then run `uv run reflex init --template blank` so the CLI is non-interactive. * Run `uv run reflex db init`, `uv run reflex db makemigrations`, and `uv run reflex db migrate` after defining the `Product` model so the schema is materialized. * Reflex's docs describe `rx.debounce_input` as a wrapper for fully controlled form controls; pass the `rx.input` as the child and bind `value=` plus `on_change=` for the search var. * Background events require `async with self:` for state mutation; do heavy DB work outside the lock to keep the UI responsive. * For the API endpoint, build a separate FastAPI `APIRouter` (or a `FastAPI` instance) and pass it to `rx.App(api_transformer=...)`. The endpoint may use a synchronous SQLAlchemy session or `rx.asession()`; the semantics must match the State logic. * Factor the SQLModel `select`/`where`/`order_by` construction into a shared helper so that the State background event and the `/api/filter` endpoint produce identical results. * Make sure `rxconfig.py` keeps the default `db_url="sqlite:///reflex.db"`; the verifier reads that file directly. ## Acceptance Criteria * Project path: `/home/user/filtered_table` * Start command (run from the project path): `uv run reflex run --backend-port 8000 --frontend-port 3000 --loglevel info` * Frontend port: 3000 (the `/` page must render with HTTP 200). * Backend port: 8000 (the `/api/filter` endpoint is mounted here via `api_transformer`). * After seeding completes, the SQLite database at `/home/user/filtered_table/reflex.db` must contain exactly 240 rows in the `product` table, distributed 40 per category for the six categories listed above, with `name`, `sku`, `price`, and `in_stock` matching the Seed Algorithm. * Source code under `/home/user/filtered_table/filtered_table/` must contain literal occurrences of all of the following Reflex idioms (verified by `grep`): * `rx.debounce_input` * `debounce_timeout=300` * `rx.asession` * `rx.foreach` * `background=True` * `api_transformer` * `GET http://localhost:3000/` returns HTTP 200 and the rendered page body references the table heading text used in the UI. * `GET http://localhost:8000/api/filter` returns the JSON object described in the API Contract, with `result_count` and `filtered` shaped as specified, and the list length matching `result_count`. * The filter logic implements the semantics described above for arbitrary combinations of the parameters, applied against the seeded data. * Background server teardown: after finishing the implementation, kill every background `reflex run` (and any other background server) you started.
